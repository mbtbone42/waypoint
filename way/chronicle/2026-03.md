# Chronicle — 2026-03

## 2026-03-26 14:00 — Project Inception and Documentation Foundation

**Type**: Decision
**Author**: Engineering
**Links**: PRD, ADR-001, ADR-002, ADR-003, DDD-001

### Context

Waypoint was conceived as a YAML-driven strategic schedule visualization tool for executive and board-level communication. The original concept and visual language come from board meetings at a startup, where a single-page timeline communicated engineering status to VCs. The format proved effective for its density-to-clarity ratio.

An initial PRD was drafted in Claude Desktop and transferred to the waypoint repository. The decision was made to follow the WAY.ai standard process before writing any code: establish the PRD, ADRs, DDD, and Chronicle foundation first.

### Narrative

The PRD was restructured from its original flat format into the WAY.ai template structure, adding explicit Goals, Non-Goals, Success Metrics, and Timeline sections while preserving all technical content. A reference image of the original board meeting format was provided to anchor the visual design.

Three architecture decisions were formalized:
- **ADR-001**: Custom minimal YAML parser (over vendoring js-yaml) — aligned with the zero-dependency philosophy
- **ADR-002**: Full SVG layout with programmatic coordinate computation — chosen for resolution independence and export quality
- **ADR-003**: SVG-to-Canvas-to-PNG export pipeline — standard browser-native approach

A single DDD was created covering the Schedule Visualization domain, with three logical sub-contexts (Data Parsing, Schedule Model, SVG Rendering) that share a single codebase given the project's small scope.

### Decision

Follow WAY.ai methodology with full documentation before implementation. Example data files will use a 14-month timeline (December 2025 through January 2027) to demonstrate realistic usage. The target audience — executives, VCs, and partners — drives the emphasis on presentation quality over interactive features.

### Rationale

Establishing the documentation foundation before code ensures architectural alignment and prevents drift. The project is small enough that the overhead is minimal, and the documents serve as a clear contract for what will be built.

### Impact

- **Created**: PRD (way/prd/overview.md), ADR-001, ADR-002, ADR-003, DDD-001, Chronicle
- **Affects**: All subsequent implementation work
- **Dependencies**: Unblocks implementation phase

## 2026-03-27 10:00 — Three-File Architecture: Data / Style / View

**Type**: Decision
**Author**: Engineering
**Links**: ADR-004, PRD (amended), DDD-001 (amended)

### Context

The original two-file architecture (data YAML + view YAML) bundled visual styling into the view file. During design review, it was recognized that style (fonts, colors, bar dimensions) and view scope (date range, track filtering) are distinct concerns. Mixing them means duplicating style declarations across every view file.

### Narrative

Three open questions from project inception were resolved:
1. **Example data**: Created `atlas-data.yaml` with a 14-month timeline (Dec 2025 - Jan 2027), four tracks (User, Features, Systems, Infrastructure), and all four status types demonstrated including two moved-out items with notes.
2. **YAML parser**: Block-only syntax confirmed — no flow sequences needed for hand-authored schedule files.
3. **Font**: Mac-first with `-apple-system, "Helvetica Neue", Arial, sans-serif`. Font specified in the style CSS file so it can be changed per-deployment. Windows font stack deferred.

The key architectural change was splitting style into its own CSS file:
- **Data YAML**: "Just the facts" — schedule content, completely independent of presentation
- **Style CSS**: "How it looks" — fonts, colors, dimensions, reusable across views
- **View YAML**: "What to show right now" — date range, track filter, title, references to data and style files

CSS was chosen over YAML for the style layer because CSS maps directly to SVG properties, is DevTools-inspectable, and supports custom properties for clean theming.

### Decision

Adopt three-file architecture per ADR-004. View files reference both data and style files via `data:` and `style:` fields. Style uses CSS custom properties for easy theming.

### Rationale

Separation of concerns: a board view and a systems deep-dive view should look the same (same style) while showing different data slices (different view). Style-as-CSS avoids inventing a custom schema for something CSS already does natively.

### Impact

- **Created**: ADR-004, atlas-data.yaml, atlas-style.css, atlas-board.yaml, atlas-systems.yaml
- **Modified**: PRD (two-file references updated to three-file), view YAML examples updated with data/style references
- **Affects**: Rendering engine must load and apply CSS; view YAML schema gains `data` and `style` fields

### Open Questions

- How to handle inline style overrides in view YAML (e.g., override a single color without a new CSS file) — defer to v2?
- Default style fallback behavior when no style file is specified

## 2026-03-27 11:00 — No Colors in Data, CDD and TSD Documents

**Type**: Decision
**Author**: Engineering
**Links**: PRD (amended), CDD, TSD, atlas-style.css (amended), atlas-data.yaml (amended)

### Context

Two refinements to the project approach:

1. The data YAML file contained `color` fields on milestones (e.g., `color: "#0066cc"`). This violates the principle that data files contain "just the facts" with no presentation concerns. Colors belong in the style CSS.

2. The way-ai process was missing two document types that are being piloted here for future inclusion: a **Code Design Document (CDD)** specifying the implementation blueprint, and a **Test Strategy Document (TSD)** defining the overall testing approach.

### Narrative

**Milestone colors moved to CSS**: Milestone `color` fields were replaced with `type` fields (e.g., `type: freeze`, `type: release`). The CSS maps these types to colors via classes (`.milestone-line.freeze`, `.milestone-label.release`). This keeps the data YAML purely factual and enables style files to define custom milestone color schemes. Three built-in milestone types: `freeze`, `release`, `deadline`. Untyped milestones use the default milestone color.

**CDD created**: The Code Design Document specifies language (vanilla JS ES modules), module breakdown (`yaml-parser.js`, `waypoint.js`, `export.js`), main data structures (Schedule model, Layout constants), entry points (`init()`, `render()`), file loading flow, and error handling. It includes a Deviation Tracking table to record where implementation diverges from the plan.

**TSD created**: The Test Strategy Document defines four test levels: unit tests for the YAML parser (highest risk), unit tests for the schedule model builder, DOM-based rendering verification, and manual testing for interactions/export. Tests use a custom in-browser test runner — no npm frameworks — consistent with zero-dependency philosophy.

### Decision

- Data YAML files must contain no color or other visual styling values. Use `type` fields that map to CSS classes.
- CDD and TSD are now part of the Waypoint docs structure. These will be proposed for inclusion in the way-ai standard process.

### Rationale

Keeping colors out of data YAML enforces clean separation at authoring time, not just at render time. A user editing the schedule should never need to think about hex codes. The CDD and TSD fill gaps in the way-ai process: CDD captures the implementation plan in a way that survives beyond a single conversation, and TSD ensures testing approach is deliberate rather than ad hoc.

### Impact

- **Created**: way/cdd/overview.md, way/tsd/overview.md
- **Modified**: atlas-data.yaml (color → type), atlas-style.css (milestone type classes added), PRD (color palette section updated, related docs updated, milestone schema updated)
- **Affects**: YAML parser does not need to handle color values; renderer maps milestone types to CSS classes

### Open Questions

- Should the CDD and TSD templates be formalized in way-ai before or after this project validates them?
- Are there other milestone types beyond freeze/release/deadline that should be pre-defined in the default CSS?

## 2026-03-27 12:00 — Dual Loading Strategy for file:// Support

**Type**: Decision
**Author**: Engineering
**Links**: ADR-005, CDD (amended), PRD (amended)

### Context

Waypoint should work when a user double-clicks `index.html` directly (file:// protocol), not just when served from an HTTP server. Browsers block `fetch()` on `file://` URLs due to same-origin policy, which breaks the standard file-loading pipeline.

### Narrative

The file:// use case matters for low-friction usage: generate YAML, double-click HTML, see the timeline. Requiring a local server adds friction, especially for non-technical users who may receive a YAML file to review.

The solution is a dual loading strategy:
1. **Embedded defaults**: Example data, view, and style are embedded directly in `index.html` as `<script type="text/yaml">` and `<style>` blocks. The tool renders immediately on file:// with no fetch.
2. **File pickers and drag-drop**: For custom data on file://, users use "Load Data/View/Style" buttons or drag files onto the page. `FileReader` API works without CORS restrictions.
3. **Protocol detection**: On init, the app checks `window.location.protocol` and adapts — HTTP mode uses `fetch()` with URL params, file mode uses embedded content and file pickers.

### Decision

Implement dual loading per ADR-005. Loading priority across both modes: drag-drop > file picker > URL params (HTTP only) > embedded defaults.

### Rationale

"Double-click and it works" is the right UX for a communication tool aimed at busy stakeholders. The cost is ~10-15KB of embedded content in `index.html` and two code paths to maintain, which is acceptable for universal accessibility.

### Impact

- **Created**: ADR-005
- **Modified**: CDD (new Dual Loading Strategy section, updated File Loading Flow), PRD (YAML Loading section rewritten as File Loading section)
- **Affects**: `index.html` structure (embedded content blocks), `waypoint.js` init logic (protocol detection, dual loading paths)

### Open Questions

- Strategy for keeping embedded defaults in sync with `examples/` directory files
- Whether to provide a build script (just a copy command) for updating embedded content, or keep it manual

## 2026-03-28 — Universal Tooltips on All Visual Objects

**Type**: Decision
**Author**: Engineering
**Links**: ADR-006, PRD (amended), CDD (amended)

### Context

Waypoint shipped with hover tooltips on item bars showing label, dates, status, and notes. In initial production use, the notes on items proved valuable for understanding context — why something slipped, what was blocking progress. The same need exists for tracks (abbreviated names need expanded descriptions), program milestones (phase objectives), and milestones (exact dates and context).

### Narrative

Stakeholders using Waypoint at work confirmed that the mouse-over tooltip pattern is the right interaction for a communication tool: the chart stays clean for quick scanning, and details are available on hover for deeper understanding.

The decision was to extend the `notes` field and hover tooltips to all visual objects:
- **Tracks**: `notes` provides expanded name or scope description, shown when hovering the rotated track label
- **Program milestones**: `notes` explains phase objectives, shown with date range and status
- **Milestones**: `notes` provides context, shown with exact date (YYYY-MM-DD)

The `notes` field is optional on all objects, so existing YAML files work unchanged.

### Decision

Add optional `notes` field to tracks, program milestones, and milestones in the data YAML schema. Add hover tooltips to track labels, program milestone bars, and milestone labels per ADR-006.

### Rationale

Progressive disclosure is the right pattern: clean chart for the 90% glance, details on demand for the 10% deep read. Adding a side panel or visible descriptions would consume chart space and violate the density-to-clarity principle. The interaction is already established on items — extending it is consistent and discoverable.

### Impact

- **Created**: ADR-006
- **Modified**: PRD (notes added to program milestone, milestone, and track schemas; Hover section expanded), CDD (data structures updated, tooltip section expanded), examples (atlas-data.yaml, embedded data in index.html), user docs (data-view-style.md)
- **Affects**: `waypoint.js` tooltip and rendering code; `buildSchedule` model builder passes through notes on all objects

## 2026-03-31 — On-Track Diamond Rule and Extended Tooltips

**Type**: Decision
**Author**: Engineering
**Links**: PRD (amended), CDD (amended)

### Context

Two changes arose from production use:

1. **On-track diamond rule**: Green diamonds were rendering on all on-track items, including those still in progress. A diamond implies a confirmed endpoint, but in-progress items don't have one. The fix: green diamonds only appear when `actual` dates exist, confirming completion. Without `actual`, the green bar ends cleanly at Today with no diamond.

2. **Extended tooltips**: After universal tooltips proved valuable on items, tracks, milestones, and program milestones, three more tooltip targets were identified:
   - **Title**: Hovering the chart title shows a project description from the view YAML (`description` field). Different views of the same data can describe the project for different audiences.
   - **Legend**: Hovering a legend entry shows what that status means. Built-in defaults ("Scheduled work that has not yet started", etc.) can be overridden via `legend_descriptions` in the view YAML — since explaining how to read the chart is a presentation concern, not a data concern.
   - **Month headers**: Hovering a month header box shows a generated summary — items completing, items starting, milestones, and program milestone activity for that month. No new YAML needed; computed from existing data at render time.

### Decision

1. Green diamonds require `actual` dates — no exceptions.
2. Title description goes in view YAML (`description` field).
3. Legend descriptions have built-in defaults, overridable via view YAML (`legend_descriptions` map).
4. Month summaries are generated from data, not authored.

### Rationale

The data/view separation guides placement: `description` and `legend_descriptions` are about how to read the chart (view concern), not schedule facts (data concern). Month summaries are derived from existing data rather than authored, avoiding duplication and keeping them automatically current.

### Impact

- **Modified**: PRD (Hover section, view YAML example, status rendering table), CDD (Schedule model, tooltip table, legend defaults, month summary generation)
- **Affects**: `waypoint.js` rendering (title tooltip, legend tooltips, month header generation), `buildSchedule` (passes through `description` and `legendDescriptions`), view YAML schema
